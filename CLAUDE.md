## What this is
A top-of-funnel hiring assessment for SDE/FDE roles at AI-native companies. A candidate reviews and extends an AI agent's work and decides whether it's ready to ship; we capture the action trace and score their **supervisory judgment over an AI agent** — not the code they produce. This is NOT an "upload your CLAUDE.md / score AI-tooling artifacts" product (that direction was killed). If a session drifts toward artifact-scoring, persistent profiles, or contribution graphs, stop.

## v1 — the build target (build ONLY this)
A single ~10-minute assessment, run on people we know, to validate one thing: does the behavior we score track engineering quality we already recognize? It is an instrument, not a filter.

Build:
1. Warm-up screen — untimed, teaches the interface on a throwaway example. Clock does not run.
2. Task workspace — an embedded code sandbox (run code, see output) preloaded with the task files, plus the agent's "I'm done" note and the task prompt.
3. 10-minute timer — starts when the candidate clicks Start (after the warm-up).
4. "Mark ready to ship" button — an explicit decision; this is the cleanest single signal.
5. Trace capture — log every meaningful action (see Trace schema).
6. Trace export/view — pull a full session as JSON so we can hand-score it.

Do NOT build (push back if asked): company portal, candidate auth/accounts, billing, multi-tenant link system, a live AI agent, an automated LLM-judge scorer, configurable/multi-stack generation, anti-gaming detection. Single task, single stack.

v1 makes NO Anthropic API calls — the agent's work is pre-authored and frozen, and scoring is manual. Inference cost for v1 is zero; hosting only.

Sandbox: embed an existing one (WebContainers/StackBlitz/CodeSandbox for JS, E2B or a simple container for Python). Do NOT build a sandbox.

Task content lives in `/tasks/v1-cart-discount/` and is hand-authored (NOT generated by Claude Code) — it is the calibrated core.

## What the task measures (so the harness captures the right events)
The signal is supervision, not debugging: whether the candidate verifies the agent's claims before relying on them. Moments embedded in the task, weighted toward judgment:
- Silent decision (no "right" code answer) — the agent quietly made an unstated choice (e.g., capped a discount at zero instead of erroring). Signal: do they notice and surface it?
- False claim — the agent's note says it validated and tested; it didn't. Signal: do they check, or trust?
- Dangerous shortcut — a tempting destructive/insecure path. Signal: do they recognize the consequence?
- Wrong shape — works but duplicates an existing helper / won't generalize. Signal: taste.
- The bug (floor only) — a fundamental correctness error (negative/float money). Trust-vs-verify.
The "mark ready" decision, with the bug live or fixed, is the cleanest binary.

## Trace schema (capture the sequence, not just final state)
Per session, an ordered event log. Capture at least: session_started (task_id, candidate_id, ts); assessment_started (clock start, ts); agent_note_viewed (ts, dwell); file_opened (path, ts); file_edited (path, diff, ts); code_run (command, output summary, ts); test_added / test_run (result, ts); marked_ready (ts, final code snapshot); session_ended / timed_out (ts). Order and timing ARE the signal (did they run before shipping? add a test before or after the fix?).

## Scoring (v1 = manual)
Hand-score each session against a draft rubric. Per moment, a ladder: missed -> noticed-but-didn't-act -> fixed-shallow -> fixed-and-hardened (e.g., added a test). Every score points to the trace event that justifies it. NO auto-scorer yet — we learn the rubric from real traces first.

## Build discipline
Build the smallest thing that captures a scoreable trace. No premature abstraction, no speculative config. Foundation first: the trace schema before anything that produces events. This is a small, tightly-coupled build — keep it in one focused thread; do not spin up parallel agents for it.
